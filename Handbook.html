Getting Started: The Universal Template

For all stations, your journey begins with the UniversalDeviceTemplate.ino file. This template is designed to give you a clear, structured starting point by handling all the complex networking behind the scenes. Your main job is to "fill in the blanks" in the designated sections with your station's specific code.

Understanding Your Template

Open the template in your Arduino IDE: File > Examples > SmartDevice > UniversalDeviceTemplate.
You will see 5 distinct sections you need to interact with:

  1. Library Inclusions:

    At the very top of the file, you'll find an import statement which tells the program to refer to our Custom <SmartDevice.h> Library. Here is where you would include any libraries you might need for your stations.
    For example, <Servo.h>, or <DHT.h>. 

    // 1. LIBRARY INCLUSIONS
#include <SmartDevice.h>
#include <Servo.h> // <-- EXAMPLE IMPORT!
  2. Variable Definitions:

    Right below the #include statements, you will find the variable definitions section. Here is where you define new variables for your program, like sensor pins, state flags, or other quantities that might need tracking.

    // 2. VARIABLE DEFINITIONS
const int sensorPin = PIN_D1; // EXAMPLE variable for a sensor pin
bool buttonState = false; // EXAMPLE variable to track button state
  3. Server Configuration:

    After defining your project variables, you'll find parameters to identify your device and its communication channels. You must change these to match your station's assignment. If your device needs access to more topics or addresses, feel free to add more topic definitions.

    // 3. CONFIGURATION
const char *DEVICE_NAME = "My_Station_Device"; // <-- CHANGE THIS! e.g., "Sarah_Door_Lock"
const char *BASE_TOPIC = "classroom/some_device"; // <-- CHANGE for your station
const char *SOME_OTHER_TOPIC = "classroom/some_other_device"; // EXAMPLE Second Topic
  4. Object Declarations:

    The final configuration section involves setting up any library objects your project requires. Objects are a library's anchor in your program and allows it to use behind-the-scenes features that drastically simplify your code. You will find a pre-defined SmartDevice object that is used to communicate with the central hub. This is pre-configured for you and you don't need to modify it. Feel free to add whatever other objects your program may require. 

    // 4. OBJECT DECLARATIONS
SmartDevice myDevice;
Servo myServo; // EXAMPLE class object to utilize servo motor
  5. Device Functions:

    Finally, you'll find the core functions that define your device's behavior. These functions are where you will write the logic for your station. The template provides three key functions that you will modify:


    1. setupDevice():

      This function runs only once when your ESP8266 first starts up. It's the perfect place to:

      • Initialize your hardware components (e.g., set pin modes, start sensors, attach servos).
      • Prepare any libraries your sensors or actuators need.

      Think of it as setting up your workstation before you begin your main project.

      void setupDevice() {
  // Your hardware setup code goes here.
  pinMode(sensorPin, INPUT); // EXAMPLE Configure sensorPin variable as an input
  pinMode(ledPin, OUTPUT); // EXAMPLE Configure ledPin variable as an output
}
    2. readSensor():

      This function runs repeatedly, many times per second, after setupDevice() is complete. It's ideal for:

      • Continuously reading data from sensors.
      • Sending periodic updates or sensor readings to the cloud.
      • Any tasks that need to happen over and over again.

      This is where you'll tell your device to keep an eye on things.

      void readSensor() {
  // Your sensor reading and data publishing logic goes here.
  int buttonState = digitalRead(PIN_D1); // EXAMPLE Read a button
  myDevice.(BASE_TOPIC, String(buttonState)); // EXAMPLE Send button state
}
    3. triggerActuator(String topic, String command):

      This function runs only when your device receives a command from the central hub (e.g., "turn light on"). It's perfect for:

      • Controlling actuators (like lights, motors, relays) based on incoming instructions.
      • Responding to specific commands with actions.
      • (For Station 6 - Master Control Panel: You will replace the default use of this function with a more advanced callback as explained in your mission briefing).

      This is where your device takes action based on external instructions.

      void triggerActuator(String command) {
  // Your actuator control logic goes here.
  if (command == "ON") { digitalWrite(PIN_D7, HIGH); } // EXAMPLE Actuation Sequence
}

Key elements handled by the System Engine include:

  • Calling your setupDevice(), readSensor(), and triggerActuator() functions at the appropriate times.

    1. Unplug your ESP8266 board.

Station 1: HVAC & Kettle Control

Your Mission: Climate & Convenience Commanders

You are the Climate & Appliance Team. Your mission is to give the central hub remote control over two high-power devices: a fan (simulating an HVAC system) and an electric kettle. You'll use special, pre-wired relay modules that act like smart switches, allowing your low-power ESP8266 to safely turn these devices on and off. This station is crucial for maintaining a comfortable and convenient smart home environment!

Real-World Application:

Think about smart thermostats like Nest or Ecobee that learn your preferences, or smart plugs that let you control any appliance from your phone. You're building the core intelligence behind such systems, enabling energy efficiency and automated comfort.

Hardware Kit

  • 1x ESP8266 Board & Breadboard
  • 1x Custom HVAC Power Plug (Relay)
  • 1x Custom Kettle Power Plug (Relay)

Wiring Guide

Relays are essentially remote-controlled switches. They have a signal pin that connects to your ESP8266. When your ESP sends a HIGH signal to this pin, the relay closes an internal switch, allowing power to flow to the connected device (turning it ON). A LOW signal opens the switch, turning the device OFF. Always ensure you connect the grounds of your ESP and relays together!

  1. HVAC Relay: Connect its Signal wire to pin PIN_D7. Connect its Ground wire to a GND pin on your ESP8266.
  2. Kettle Relay: Connect its Signal wire to pin PIN_D6. Connect its Ground wire to another GND pin on your ESP8266.

Coding Mission: Step-by-Step

Your station acts as an actuator—it reacts to commands. You will primarily write code in the setupDevice() and triggerActuator() functions within your UniversalDeviceTemplate.ino.

  • Configuration: Update the DEVICE_NAME and BASE_TOPIC in your template. Then, subscribe to the specific command topics for HVAC and Kettle. 
    // 3. SERVER CONFIGURATION
        const char* DEVICE_NAME = "YourName_HVAC_Kettle"; // e.g., "Sarah_HVAC_Kettle"
        const char* BASE_TOPIC = "classroom/hvac_kettle"; // <-- CHANGE THIS!
  • Define Pins (Recommended): After the #include <SmartDevice.h> line, define constants for your pins to make your code more readable: 
    #define HVAC_RELAY_PIN PIN_D7
        #define KETTLE_RELAY_PIN PIN_D6
  • setupDevice(): In this function, you'll configure your relay pins as outputs and ensure they start in a safe (OFF) state. Also, subscribe to the command topics. 
    void setupDevice() {
          // Configure both relay pins as outputs
          pinMode(HVAC_RELAY_PIN, OUTPUT);
          pinMode(KETTLE_RELAY_PIN, OUTPUT);
          // Ensure both relays start OFF to prevent unexpected activations
          digitalWrite(HVAC_RELAY_PIN, LOW);
          digitalWrite(KETTLE_RELAY_PIN, LOW);
        
          // Subscribe to the command topics for HVAC and Kettle
          myDevice.subscribeTo(String(BASE_TOPIC) + "/hvac");
          myDevice.subscribeTo(String(BASE_TOPIC) + "/kettle");
        }
  • triggerActuator(String topic, String command): This is where you'll write the logic to control your relays based on the incoming topic and command string. Remember to report the status back to the cloud using myDevice.publishTo(). 
    void triggerActuator(String topic, String command) {
          Serial.print("Received on topic: ");
          Serial.print(topic);
          Serial.print(". Payload: ");
          Serial.println(command);
        
          // Handle HVAC commands based on its topic
          if (topic == String(BASE_TOPIC) + "/hvac") {
            if (command == "ON") {
              digitalWrite(HVAC_RELAY_PIN, HIGH);
              myDevice.publishTo(String(BASE_TOPIC) + "/hvac", "ON");
              Serial.println("HVAC ON");
            } else if (command == "OFF") {
              digitalWrite(HVAC_RELAY_PIN, LOW);
              myDevice.publishTo(String(BASE_TOPIC) + "/hvac", "OFF");
              Serial.println("HVAC OFF");
            }
          }
          // Handle Kettle commands based on its topic
          else if (topic == String(BASE_TOPIC) + "/kettle") {
            if (command == "ON") {
              digitalWrite(KETTLE_RELAY_PIN, HIGH);
              myDevice.publishTo(String(BASE_TOPIC) + "/kettle", "ON");
              Serial.println("Kettle ON");
            } else if (command == "OFF") {
              digitalWrite(KETTLE_RELAY_PIN, LOW);
              myDevice.publishTo(String(BASE_TOPIC) + "/kettle", "OFF");
              Serial.println("Kettle OFF");
            }
          }
        }

Success Criteria

Your mission is a success when you can independently control both the fan (HVAC) and the kettle from the central dashboard or via voice commands, and their ON/OFF status updates correctly on the dashboard. Ask your instructor to test it!

Challenges & Tips:
  • Initial State: Ensure your relays are OFF when the device first starts to prevent unexpected appliance activation. You can do this in setupDevice().
  • Debugging: Use Serial.println() statements in your triggerActuator function to see what payloads you're receiving and confirm your logic is being executed.
  • Power Considerations: Relays can sometimes draw a bit more current than regular LEDs. Make sure your ESP8266 is powered correctly (usually via USB).

Station 2: Environmental & Security Sensors

Your Mission: The Classroom's Senses

You are the Sensor Integration Team. Your task is to become the "eyes and ears" of the Smart Classroom by continuously monitoring three different types of sensors: a DHT11 for precise temperature readings, a PIR sensor for motion detection, and a magnetic contact switch to detect if a window is open or closed. Your data will provide the hub with crucial information for automated responses!

Real-World Application:

This is how smart homes know what's going on! Think about intelligent climate control that reacts to temperature changes, security systems that detect intruders with motion and door sensors, or even smart ventilation systems. Your work here is fundamental to an intelligent building.

Hardware Kit

  • 1x ESP8266 Board
  • 1x DHT11 Temperature
  • 1x PIR Motion Sensor
  • 1x Magnetic Contact Switch (for window)

Wiring Guide

Each sensor connects slightly differently, but the principles of power, ground, and a signal pin remain. The DHT11 needs power and a single data line. The PIR sensor is similar. The contact switch is the simplest; it's just a switch that completes a circuit when open or closed.

  1. DHT11: Connect its Data pin to PIN_D2, VCC/Vin (positive power) to 3V3, and GND (ground) to GND on your ESP8266.
  2. PIR Sensor: Connect its OUT (signal) pin to PIN_D5, VCC to 3V3, and GND to GND.
  3. Window Sensor (Magnetic Contact Switch): This is a simple switch. Connect one wire of the switch to PIN_D1 and the other wire to GND. We will use the ESP's special internal pull-up resistor for this.

Coding Mission: Step-by-Step

Your station uses sensors—it senses the world and reports data. You will primarily write code in the setupDevice() and readSensor() functions within your UniversalDeviceTemplate.ino. The triggerActuator() will not be used by this station.

  • Configuration & Libraries: Update the DEVICE_NAME and `BASE_TOPIC`. Also, add the DHT library and define your sensor objects and pins: 
    #include <DHT.h>
        
        const char* DEVICE_NAME = "YourName_EnviroSensor"; // e.g., "Alex_EnviroSensor"
        const char* BASE_TOPIC = "classroom/environment"; // <-- CHANGE THIS!
        
        // DHT sensor setup
        #define DHTPIN PIN_D2
        #define DHTTYPE DHT11
        DHT dht(DHTPIN, DHTTYPE);
        
        #define PIR_PIN PIN_D5
        #define WINDOW_PIN PIN_D1
  • setupDevice(): Initialize your DHT sensor and set pin modes for the PIR and window sensors. 
    void setupDevice() {
          dht.begin(); // Start the DHT sensor.
          pinMode(PIR_PIN, INPUT); // PIR sensor output is an input to the ESP.
          pinMode(WINDOW_PIN, INPUT_PULLUP); // Use internal pull-up resistor for window sensor.
        }
  • readSensor(): This is where you will continuously read all your sensors, combine the data into the required string formats, and publish them to the cloud. 
    void readSensor() {
          // This function runs repeatedly to read sensors and send data.
          // Read temperature from DHT11.
          float temperatureC = dht.readTemperature();
        
          // Read motion detection from PIR sensor.
          bool motionDetected = (digitalRead(PIR_PIN) == HIGH);
        
          // Only publish if DHT readings are valid.
          if (!isnan(temperatureC)) {
            myDevice.publishTo(String(BASE_TOPIC) + "/temperature", String(temperatureC));
            Serial.print("Temperature sent: "); Serial.println(temperatureC);
          } else {
            Serial.println("Failed to read from DHT sensor!");
          }
        
          // Publish motion status.
          myDevice.publishTo(String(BASE_TOPIC) + "/motion", motionDetected ? "MOTION_DETECTED" : "NO_MOTION");
          Serial.print("Motion status sent: "); Serial.println(motionDetected ? "MOTION_DETECTED" : "NO_MOTION");
        
          // Read window sensor status and publish separately.
          // LOW means the switch is closed (magnet is near), indicating the window is CLOSED.
          bool isWindowClosed = (digitalRead(WINDOW_PIN) == LOW);
          myDevice.publishTo(String(BASE_TOPIC) + "/window", isWindowClosed ? "CLOSED" : "OPEN");
          Serial.print("Window status sent: "); Serial.println(isWindowClosed ? "CLOSED" : "OPEN");
        }

Success Criteria

Your mission is a success when the central dashboard shows live, updating values for temperature, motion detection, and the window's open/closed status. You can even ask Google Assistant, "Hey Google, what's the temperature in the classroom?"

Challenges & Tips:
  • DHT Errors: Sometimes, DHT sensors can give NaN (Not a Number) if they fail to read. Add if (isnan(temp)) checks and skip publishing if data is bad.
  • String Concatenation: Building topics like `String(BASE_TOPIC) + "/temperature"` is important. Practice converting numbers and booleans into text using `String()` and combining them with `+`.
  • Efficiency: Avoid sending data too frequently. The delay(1000) in the main loop (handled by the template's "System Engine") helps, but be mindful of how often you read and publish.

Station 3: Smart Lighting System

Your Mission: The Light Engineers

You are the Lighting Engineering Team. Your challenge is to control two different lights: a powerful, multi-color RGB LED strip for the main room lighting, and a simple LED for an outdoor "porch light." This mission involves not just turning lights on and off, but also parsing complex commands from the hub to control colors and brightness!

Real-World Application:

This is the heart of modern smart lighting, like Philips Hue or LIFX systems! You're learning how to create dynamic lighting scenes, adjust ambiance, and even potentially automate lighting based on time of day or occupancy, contributing to energy savings and user comfort.

Hardware Kit

  • 1x ESP8266 Board
  • 1x Addressable RGB LED Strip (NeoPixel or WS2812B compatible)
  • 1x Regular LED & 1x 220 Ohm Resistor (to protect the LED)

Wiring Guide

The RGB Strip (NeoPixel) requires its own external power source because it can draw a lot of current (ask your instructor for guidance on connecting this!). However, it receives its data signal from one of your ESP8266 pins. The simple LED for the porch light is powered directly from an ESP pin, but always use a resistor to limit the current and protect both the LED and your ESP!

  1. RGB Strip (NeoPixel): Connect the strip's Data In (often labeled DIN or DI) to pin PIN_D4. Connect the strip's 5V (positive power) and GND (ground) to the separate external power supply as directed by your instructor. CRITICAL: You MUST also connect the strip's GND to a GND pin on your ESP8266 to ensure both devices share a common ground reference.
  2. Porch Light LED: Connect the long leg (anode, positive) of the LED to one end of the 220Ω resistor. Connect the other end of the resistor to pin PIN_D7. Connect the short leg (cathode, negative) of the LED to a GND pin on your ESP8266.

Coding Mission: Step-by-Step

Your station acts as an actuator—it reacts to commands. You will primarily write code in the setupDevice() and triggerActuator() functions within your UniversalDeviceTemplate.ino. The readSensor() will not be used by this station.

  • Configuration & Libraries: Update the DEVICE_NAME and `BASE_TOPIC`. Also, include the Adafruit NeoPixel library and define your NeoPixel object and pins: 
    #include <Adafruit_NeoPixel.h>
        
        const char* DEVICE_NAME = "YourName_Lighting"; // e.g., "Chris_Lighting"
        const char* BASE_TOPIC = "classroom/lighting"; // <-- CHANGE THIS!
        
        #define NEOPIXEL_PIN PIN_D4
        #define NUM_PIXELS 12 // Adjust based on your strip's actual length
        
        Adafruit_NeoPixel strip(NUM_PIXELS, NEOPIXEL_PIN, NEO_GRB + NEO_KHZ800); // Correct constructor order
        
        #define PORCH_LIGHT_PIN PIN_D7
  • setupDevice(): Initialize the NeoPixel strip, set its initial state, and configure the porch light pin. Subscribe to the command topics. 
    void setupDevice() {
          strip.begin();
          strip.show(); // Initialize all pixels to 'off'
          strip.setBrightness(50); // Start with a moderate brightness
        
          // Configure porch light pin
          pinMode(PORCH_LIGHT_PIN, OUTPUT);
          digitalWrite(PORCH_LIGHT_PIN, LOW); // Ensure porch light starts OFF
        
          // Subscribe to command topics for RGB strip and porch light
          myDevice.subscribeTo(String(BASE_TOPIC) + "/rgb");
          myDevice.subscribeTo(String(BASE_TOPIC) + "/porch_light");
        }
  • triggerActuator(String topic, String command): This is the core of your lighting control. You'll need to parse commands for both the simple porch light and the more complex RGB strip. 
    void triggerActuator(String topic, String command) {
          Serial.print("Received on topic: ");
          Serial.print(topic);
          Serial.print(". Payload: ");
          Serial.println(command);
        
          // Handle Porch Light commands (simple ON/OFF)
          if (topic == String(BASE_TOPIC) + "/porch_light") {
            if (command == "ON") {
              digitalWrite(PORCH_LIGHT_PIN, HIGH);
              myDevice.publishTo(String(BASE_TOPIC) + "/porch_light", "ON");
              Serial.println("Porch Light ON");
            } else if (command == "OFF") {
              digitalWrite(PORCH_LIGHT_PIN, LOW);
              myDevice.publishTo(String(BASE_TOPIC) + "/porch_light", "OFF");
              Serial.println("Porch Light OFF");
            }
          }
          // Handle Main RGB Strip commands. Utilize SmartDevice::commandToLight for parsing.
          else if (topic == String(BASE_TOPIC) + "/rgb") {
            SmartHome::Light lightState = SmartDevice::commandToLight(command);
            
            if (lightState.isOn) {
              // Set brightness for the strip (0-255 scaling)
              strip.setBrightness(map(lightState.brightness, 0, 100, 0, 255));
              // Set color for all pixels on the strip
              for(int i=0; i < NUM_PIXELS; i++) {
                if (lightState.type == SmartHome::COLOR) {
                  // Convert HSL to RGB for NeoPixel, example requires custom function or library
                  // For simplicity, direct RGB values are often sent, but this structure uses HSL
                  // For now, if COLOR type, we'll assume a default color or implement HSL->RGB conversion.
                  // Example: pixels.setPixelColor(i, pixels.ColorHSV(lightState.hue * 182, lightState.saturation * 255 / 100, lightState.brightness * 255 / 100));
                  strip.setPixelColor(i, 255, 255, 255); // Placeholder: set white if color command is complex
                } else {
                  strip.setPixelColor(i, 255, 255, 255); // Default white for dimmable
                }
              }
              strip.show();
              Serial.println("RGB Strip ON");
            } else {
              strip.clear(); // Turn all pixels off
              strip.show();
              Serial.println("RGB Strip OFF");
            }
            myDevice.publishTo(String(BASE_TOPIC) + "/rgb", lightState.payload());
          }
        }

Success Criteria

You've succeeded when you can ask Google Assistant to change the main room light's color (e.g., "turn main light blue"), adjust its brightness, and separately turn the porch light on and off, with all commands responding reliably!

Challenges & Tips:
  • NeoPixel Library: Make sure you install the `Adafruit NeoPixel` library via the Arduino IDE's Library Manager. The `SmartDevice::commandToLight` function returns a `SmartHome::Light` struct with `hue`, `saturation`, and `brightness` values. You'll need to convert these to `Adafruit_NeoPixel`'s `pixels.Color()` or `pixels.ColorHSV()` format.
  • Powering NeoPixels: If your NeoPixels behave erratically or show wrong colors, it's often a power issue. Ensure your external power supply is sufficient and grounds are properly connected.
  • Serial Debugging: Print the `topic` and `command` in `triggerActuator` to verify you are receiving the correct instructions from the hub.

Station 4: Smart Door Security

Your Mission: Guardians of the Gateway

You are the Security & Access Team. Your critical mission is to build a complete smart door system. You'll use two servo motors (inverted to each other to simulate a robust locking mechanism) and a magnetic contact switch to detect if the door is open or closed. You must constantly report the complete status of both the door (open/closed) and the lock (locked/unlocked) to the central hub.

Real-World Application:

This is exactly how smart locks (like August or Yale Smart Lock) work! You're building the core components of a sophisticated access control system that can be remotely managed and provide real-time security alerts. Imagine integrating this with security cameras or smart alarms!

Hardware Kit

  • 1x ESP8266 Board
  • 2x Servo Motor (e.g., SG90 or similar, often miniature)
  • 1x Magnetic Contact Switch

Wiring Guide

Servos can draw more power than a digital pin can directly supply, so we power them from the VIN pin on your ESP8266 board, which provides a steady 5V directly from the USB connection. The magnetic contact switch works just like the one in Station 2, using the ESP's internal pull-up resistor.

  1. Servo 1 (Lock Mechanism): Connect its Power (red wire, usually) to the VIN pin on your ESP8266. Connect its Ground (brown/black wire) to a GND pin. Connect its Signal (orange/yellow wire) to PIN_D6.
  2. Servo 2 (Inverted Lock Mechanism): Connect its Power to VIN and Ground to GND. Connect its Signal to PIN_D5. This servo will move in the opposite direction of Servo 1 to ensure a robust lock/unlock action.
  3. Door Sensor (Magnetic Contact Switch): Connect one wire of the switch to PIN_D1 and the other wire to GND. We will configure this pin with INPUT_PULLUP mode in your code.

Coding Mission: Step-by-Step

Your station acts as both an actuator (lock) and a sensor (door state). You will primarily write code in the setupDevice(), readSensor() and triggerActuator() functions within your UniversalDeviceTemplate.ino.

  • Configuration & Libraries: Update the DEVICE_NAME and `BASE_TOPIC`. Also, include the Servo library, define your servo objects and pins, and declare a global variable to track the lock state: 
    #include <Servo.h>
        
        const char* DEVICE_NAME = "YourName_Door"; // e.g., "Emily_Door"
        const char* BASE_TOPIC = "classroom/door_security"; // <-- CHANGE THIS!
        
        Servo lockServo1;
        Servo lockServo2;
        
        #define SERVO_PIN_1 PIN_D6
        #define SERVO_PIN_2 PIN_D5
        #define DOOR_SENSOR_PIN PIN_D1
        
        bool isLocked = false; // Global variable to track the current lock state
  • setupDevice(): Initialize your servos and set the pin mode for the door sensor. Subscribe to the command topic. 
    void setupDevice() {
          lockServo1.attach(SERVO_PIN_1);
          lockServo2.attach(SERVO_PIN_2);

Station 5: Automated Curtain

Your Mission: The Mechatronics Maestros

You are the Mechatronics Team. This is a low-level control challenge that combines mechanical movement with smart logic! You'll program the intelligence for a motorized smart curtain. You will use a positional servo as the motor and two photo-interrupter switches to detect when the curtain is fully open or fully closed. Real-World Application:

Think about automated window shades in smart homes or offices that can adjust based on sunlight, time of day, or privacy needs. You're building the core control system for such convenience, integrating mechanical components with intelligent software.

Hardware Kit

  • 1x ESP8266 Board
  • 1x Positional Servo Motor (to simulate curtain movement)
  • 2x Photo-interrupter Switches (one for "fully open", one for "fully closed")

Wiring Guide

A positional servo is wired just like a standard servo. The photo-interrupter switches are sensors that detect when something breaks a light beam; they will be physically triggered when the curtain reaches its fully open or closed position.

  1. Positional Servo: Connect its Power (red wire) to VIN (5V from USB), its Ground (black/brown wire) to GND, and its Signal (yellow/orange wire) to PIN_D6.
  2. Photo-interrupter Switches: These switches have a signal output. Connect the signal pin of the "Fully Open" switch to PIN_D1 and the "Fully Closed" switch to PIN_D2. You will likely need to supply 3.3V and GND to these sensors as well (check datasheet).

Coding Mission: Step-by-Step

This station involves controlling a motor and reading sensors to manage the curtain's movement. You will primarily write code in the setupDevice(), readSensor() and triggerActuator() functions within your UniversalDeviceTemplate.ino.

          Serial.print(topic);
          Serial.print(". Payload: ");
          Serial.println(command);

              myCurtainMotor.write(180); // Adjust angle for fully open position
              currentCurtainState = "MOVING"; // Set state to moving
              Serial.println("Curtain OPENING");
            } else if (command == "CLOSE") {
              myCurtainMotor.write(0); // Adjust angle for fully closed position
              currentCurtainState = "MOVING"; // Set state to moving
              Serial.println("Curtain CLOSING");
            } else if (command == "STOP") {
              myCurtainMotor.write(myCurtainMotor.read()); // Stop at current position (requires a servo that holds position)
              currentCurtainState = "STOPPED"; // Indicate stopped state
              Serial.println("Curtain STOPPED");
              myDevice.publishTo(String(BASE_TOPIC) + "/state", "STOPPED");
            }
          }
        }
  • readSensor(): This is where the state machine logic lives. It will continuously check the current state (from the switches) and publish status updates. You might also want to stop the servo once a limit switch is hit. 
    void readSensor() {
          // Read the state of the open and closed photo-interrupter switches.
          // Assuming LOW means the beam is interrupted (curtain is at that position).
          bool openSensorActive = (digitalRead(OPEN_SENSOR_PIN) == LOW);
          bool closedSensorActive = (digitalRead(CLOSED_SENSOR_PIN) == LOW);
        
          // Update internal state and publish if state changes
          String newCurtainState = currentCurtainState;
        
          if (openSensorActive && newCurtainState != "OPEN") {
            newCurtainState = "OPEN";
            myCurtainMotor.write(myCurtainMotor.read()); // Stop the servo if it hits open limit
          } else if (closedSensorActive && newCurtainState != "CLOSED") {
            newCurtainState = "CLOSED";
            myCurtainMotor.write(myCurtainMotor.read()); // Stop the servo if it hits closed limit
          } else if (!openSensorActive && !closedSensorActive && (currentCurtainState == "OPEN" || currentCurtainState == "CLOSED")) {
            // If it was open/closed but now neither sensor is active, it must be moving (unless commanded STOPPED).
            // This is a simplified state; for robust systems, manage this with command.
            newCurtainState = "MOVING";
          }
        
          if (newCurtainState != currentCurtainState) {
            currentCurtainState = newCurtainState;
            myDevice.publishTo(String(BASE_TOPIC) + "/state", currentCurtainState);
            Serial.print("Curtain status updated: ");
            Serial.println(currentCurtainState);
          }
        }

Success Criteria

Success is when the curtain opens and closes fully via voice commands, stopping automatically and reliably when it hits the respective photo-interrupter switches (both open and closed). The central dashboard should also accurately reflect the curtain's state as "OPEN" or "CLOSED" when it reaches an end point, or "STOPPED" if you manually stop it.

Challenges & Tips:
  • Photo-interrupter Logic: Understand if your photo-interrupter outputs HIGH or LOW when the beam is broken, and adjust your `digitalRead` checks accordingly. `INPUT_PULLUP` will make the pin HIGH when the beam is NOT broken, and LOW when it IS broken (which is often what you want for limit switches).
  • Positional Servo: Unlike continuous rotation, a positional servo moves to a specific angle and stops. You'll need to map your "OPEN" and "CLOSE" commands to specific angles (0-180 degrees) for your servo and physical curtain setup.
  • Robustness: Consider what happens if a photo-interrupter fails. How can you add a timeout to prevent the motor from running indefinitely? (These are advanced questions for later!)
  • Debugging States: Print the `currentCurtainState` variable to your `Serial Monitor` in the `loop()` to observe how your state machine is transitioning. This is a powerful debugging technique.

2. Code Logic & Syntax Errors

These are errors in your C++ code that prevent it from compiling or running correctly. The Arduino IDE compiler will give you feedback.

  • Compiler Errors: When you click "Verify" or "Upload," the bottom pane of the Arduino IDE will show red error messages. Read these carefully! They often tell you the file name and line number where the error occurred. Common errors include missing semicolons (;), misspelled variable/function names, unmatched parentheses (), curly braces {}, or square brackets [].
  • Incorrect Variable/Function Scope: Ensure variables and functions are declared in the correct places (e.g., global variables outside of functions, local variables inside the function where they are used).
  • Case Sensitivity: C++ is case-sensitive. mydevice is different from myDevice. Always match the exact casing.
  • Library Issues: If you include new libraries (e.g., #include <DHT.h>), ensure they are installed via the Arduino IDE's Library Manager (Sketch > Include Library > Manage Libraries...).
  • Logic Bugs (Runtime Errors): Your code compiles but doesn't do what you expect. Use Serial.print() and Serial.println() extensively in your setupDevice(), readSensor(), and triggerActuator() functions to print variable values and track program flow. This is your most powerful debugging tool.

3. Upload / Port Issues

Problems getting your compiled code onto the ESP8266 board.

  • Incorrect Board Selection: Go to "Tools > Board" and ensure "NodeMCU 1.0 (ESP-12E Module)" or the equivalent for your board is selected.
  • Incorrect Port Selection: Go to "Tools > Port" and select the correct COM port for your ESP8266. If you unplug and replug the USB cable, the port might change. On Windows, this is usually COMx; on macOS, it might be /dev/cu.usbserial-XXXX.
  • Driver Issues: For some ESP8266 boards, you might need a USB-to-Serial driver (e.g., CH340G or CP2102 driver). If no COM port appears even after plugging in, you likely need to install the correct driver for your board's USB chip.
  • "Failed to connect / A fatal error occurred" during Upload:
    • Press and Hold FLASH button: Some boards require you to press and hold the "FLASH" or "BOOT" button on the ESP8266 board while uploading, then release it after the upload starts (you'll see "Connecting..." in the IDE).
    • Loose USB Cable: Ensure your USB cable is fully inserted and not faulty. Try a different cable if possible.
    • Power Issues: Ensure your board is adequately powered. USB 2.0 ports generally provide enough power, but sometimes a powered USB hub or a shorter, higher-quality cable can help.
  • Conflicting Programs: Ensure no other program (like another Serial Monitor, or other IDE) is currently using the COM port. Close any such applications.

4. Physical Wiring Mistakes

Even perfect code won't work with incorrect wiring. Always double-check your physical connections!

  • Component Orientation: Many components (LEDs, sensors, chips) are polarized or have specific pin orientations. Ensure they are inserted correctly. Look for small dots, notches, or longer/shorter legs.
  • Correct Pin Usage: Verify that you are connecting components to the exact GPIO pins specified in your code (e.g., if code says PIN_D1, connect to the physical D1 pin on the NodeMCU). Refer to the GPIO Pin Definitions in the API Reference section.
  • Loose Connections: Wires can easily come loose on breadboards or header pins. Gently push all wires to ensure a firm connection.
  • Short Circuits: Ensure no bare wires or component leads are touching unintentionally, which can cause unexpected behavior or damage components.
  • Power and Ground: Always ensure your components have a common ground connection with the ESP8266, and that power (3.3V or 5V, depending on component) is correctly supplied.
